## 2. 我的理解（基于 输入 2 修订）

### 三页核心
- **介绍页（intro）**：纯介绍、打字机一行行（SSE 风格、可换行）、点击跳过当前页/翻下一页、第二页起左上「←」、可被「关于游戏」重新触发。
- **菜单页（home / entry）**：三按钮居中。
  - `[开始新故事]`：左侧滑出**剧本**列表；选中剧本 → 中央以打字机展示该剧本简介 → 「开始新游戏」/「继续游戏」
  - `[档案]`：左侧滑出**剧本**列表；选中剧本 → 二级侧边栏滑出**该剧本所有存档**（自动 + 手动）→ 点击进入读档
  - `[关于游戏]`：跳回介绍页重播
- **故事页（game）**：单屏面板 + 底部居中输入框（Enter 提交）。`view` 三态：`terminal` / `doc` / `rules`。每条命令刷新单屏，不拼接。

### 「剧本/故事/存档」的关系（用户重申）
```
剧本（story）── 多段剧情 ── 多份文档（archive）
              └── 多份存档（saves）─── 每个剧本独立存档命名空间
```

### 存档体验需要做到
- 自动存档（每个剧本一份固定 slot）
- 手动存档（多份，可命名，按时间倒序）
- 自动不会覆盖手动；从手动存档读档后，自动 slot 从该点继续
- 支持删除存档、重命名存档
- UI 上能直观看到「自动存档」和「手动存档」的分组

### `back` 限定语义
- 仅在 `menuMode === "game"` 时有效
- 维护一个游戏屏幕栈（gameStack），`setScreen` 时 push 当前；`back` 命令 pop
- 出栈到底了就提示「无更多可返回的档案屏」

### 6.6 教训记录（值得未来沿用）
- **SugarCube widget 中 HTML 属性插值不能用 `<<= ... >>`**，必须用 `@attr="expr"` 属性指令，否则属性值会保留字面 wiki 文本——而且不会报错，bug 会潜伏到运行时数据被污染（这次表现为 library key 为 `<<= $selectedStory >>`）
- 文本节点内 `<<= ... >>` 是 OK 的，区别在 attribute boundary
- widget 内复杂 JS 表达式（function literal、find/filter）建议提取为 `setup.*` 辅助函数，wikifier 解析更稳

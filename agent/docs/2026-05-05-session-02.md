# Session 2026-05-05 · 命令体系重构 + 沉浸感升级（第 02 次）

## 1. 用户输入（要点提取）

> 「help 后没有指令的可选参数配置说明，例如 save 没有提示可以指定名字」
> 「没有逐步解锁指令的感觉」
> 「需要可以点指令自动在输入框中输入对应的指令」
> 「load 完全没有效果」
> 「文档命名格式有问题，不是 ::，而是只有一个 :」
> 「find 命令不是我描述的效果」
> 「rule 逻辑似乎有 bug，UI 不好看」
> 「整体没有 type help 的沉浸感」
> 「back 应该返回非异常时的屏幕，例如 1.help → 2.asda → 3.error → 4.back，应该回到 help。然后在 error 时点击页面，应该恢复到 help 并把 asda 预填到输入框」
> 「整体应该更像在操作电脑，多一些游戏性辅助：存档、回到主菜单、按钮回退上一级」

## 2. 我的理解

- **指令是这游戏的主交互**，列表里要有清楚的 USAGE（参数、说明、可示例），且**所有列表都可以点击**填到输入框
- **find** 是"查已读"工具，命中已读档案时解锁该档案 `links` 里的下一级；不是上帝视角搜全部
- **load** 不带参 = 列出存档；带参 = 加载
- **逐步解锁**：command 分 `core / save / note / rule / link` 类；core 永久可见，其它通过读档案 (`unlocksCommands` 字段) 解锁。help 只列已解锁
- **back** 跳过 `error` 屏；**error 屏可点击恢复**（跳到非 error 上一屏 + 把刚才输错的命令回填到输入框）
- **加导航按钮**：游戏内左上 ← 主菜单 / ← 上一屏，让用户除指令外有图形化退路
- **沉浸感**：terminal-card 用 monospace、`> command` 提示符；rule UI 也走暗色 terminal 风
- **档案/存档列表分隔符** `::` → `:`，更"机房"

### 玩家侧行为变化
- **命令体系**：help 不再是堆字符串列表，而是一行一行的 chip：左边 chip 是命令名（点击 → 自动填到输入框），右边一句话说明
- **逐步解锁**：开局只有 core 命令；读邀请函解锁 save；读《校园管理准则》解锁 note / rule / link
- **find 改语义**：仅在已恢复档案中搜，命中档案的关联档案被恢复（带"新恢复"标签），未命中给出引导提示
- **load 列档案**：load 无参列出全部存档；load `<id|名字>` 直接加载
- **存档 / 档案分隔符**：`::` 全部改 `:`
- **back 跳错**：错命令 → error 屏 → back，跳过 error 直接回到上一个非错误屏
- **error 屏点击**：error 卡片本身可点（红虚线 + cursor:pointer + 底部提示），点击后 back 一次并把刚才输错的命令字面回填到输入框（继续编辑而不是重打）
- **顶部导航按钮**：游戏内左上角浮动两个按钮，一个 ← 主菜单（先 autoSave），一个 ← 上一屏
- **沉浸感**：终端卡用 monospace + 终端绿色提示符；规则填写表也走相同的暗色风格

### 教训记录
- SugarCube `<<if>>` 内可以用 `typeof X === "string"`（标准 JS 表达式）做分支
- screenLines 是个开放容器，让它接受字符串和对象两种元素是渐进改造的关键，避免一刀切重构所有 setScreen 调用
- 加 click handler 时务必在容易冲突的位置（嵌套容器）调 `ev.stopPropagation()` 或在外层 handler 里 closest-check，避免 chip 点击冒泡到 .terminal-error 触发 back

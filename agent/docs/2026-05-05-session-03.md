## 2. 我的理解

- 用户希望**剧情作者用 GUI 改档案文字**，但保持系统层（widget / 命令逻辑 / CSS）在 .twee 源
- 当前档案正文（`archive[i].body`）写死在 `setup.archive` 的 JS 数组里 → GUI 完全看不到
- 解决：把每份档案的 body 抽到独立的 `:: archive_<id> [archive]` passage；setup.archive 只留 metadata + 一个 `passage: "archive_<id>"` 字段
- 系统调用从 `doc.body`（数组字段）改为 `setup.docBody(doc)`（运行时从 passage 文本读取）
- README 加 GUI 工作流（Twine 网页版 import dist/index.html → 配 SugarCube format → 编辑 [archive] 节点 → 桌面版 export to twee 同步回源）

## 6. 教训

- **数据 vs 文本要分层**：keywords / links / unlocksCommands 这种程序逻辑用得到的字段必须留在 JS 数据里；正文/标题这种"读出来的"是文本，应该独立。这次的拆法保留了这层分隔
- **passage 即源**：用 `Story.get(name).text` 在 SugarCube 运行时读 passage 原文是稳定的（不被 wikifier 干扰），适合做"档案正文池"
- **widget 的健壮性**：docViewer 之前依赖 `_doc.body` 字段，迁移过程中如果 setup.archive 拿掉这个字段而 widget 没改，会静默渲染空内容（passage 加载没问题，commandInput 也渲染——但用户看到空档案）。这次靠 smoke test 命令 input 不可见暴露了渲染崩溃，说明 `<<for>> undefined` 在 Sugar wikifier 中会让整个 passage 渲染卡住——值得记一笔
- **README 是流程文档**：剧情作者来了第一次怎么操作，应该一步一步写出来，包括 SugarCube format URL 这种容易踩坑的 host 路径
